# Task: i18n 시스템 고도화 및 전역 스토어 보안 아키텍처 개편

## 무엇을 했는가?
애플리케이션 전반의 i18n(국제화) 시스템을 리액트 훅 기반(`useT`)으로 최신화하고, Zustand 스토어를 싱글톤 패턴에서 **Provider 패턴(Context)**으로 전면 개편했습니다. 또한 서버 사이드 렌더링(SSR) 시 발생하는 상태 오염(State Pollution) 문제를 해결하기 위해 요청별 독립 스토어 구조를 구축했습니다.

## 왜 그렇게 했는가?
1. **사용자 경험(UX) 최적화**: 페이지 로드 직후 번역 키가 잠시 노출되는 현상을 제거하기 위해 SSR 단계에서 미리 번역 데이터를 시딩해야 했습니다.
2. **보안 및 SSR 무결성 확보**: Next.js App Router의 서버 환경에서 싱글톤 스토어를 사용하면 유저 간 세션이나 개인 정보가 섞일 위험이 있어, 요청마다 격리된 스토어 인스턴스가 필수적이었습니다.
3. **코드 일관성 및 유지보수**: `useI18n`을 `useT`로 개편하여 코드를 간결하게 하고, 모든 스토어를 `StoreProvider`라는 통합된 진입점에서 관리하도록 아키텍처를 표준화했습니다.

## 기술적 결정 사항
- **Unified StoreProvider**: `App`, `User`, `Market`, `UI` 스토어를 하나의 컨텍스트 레이어로 통합하여 `layout.tsx`를 정리하고 하이드레이션 순서를 보장했습니다.
- **Factory Pattern (`createStore`)**: 스토어 정의 시 `create` 대신 `createStore`를 사용하여 클라이언트와 서버 모두에서 요청별 독립성을 보장하는 객체를 생성하도록 했습니다.
- **Static Store Bridge**: `ui.modal` 같이 훅을 사용하기 어려운 정적 유틸리티에서 스토어에 접근할 수 있도록 클라이언트 측 전용 브릿지(`setUIStore`)를 구현했습니다.

## 결과
- SSR 단계에서 완성된 HTML이 전달되므로 초기 로딩 시 번역 키 노출 현상이 완벽하게 해결되었습니다.
- 모든 전역 상태가 요청별로 격리되어 유저 간 데이터 혼선 및 누수 위험이 제거되었습니다.
- `npm run lint` 및 `npm run build`를 통해 코드 품질과 타입 안정성을 최종 확보했습니다.
