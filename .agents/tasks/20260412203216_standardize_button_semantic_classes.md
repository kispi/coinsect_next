# TASK: 버튼 컴포넌트 표준화 및 세맨틱 클래스 도입

## WHAT
- 하드코딩된 Tailwind 유틸리티 클래스 대신 세맨틱 CSS 클래스(`btn-primary`, `btn-default` 등) 도입
- `globals.css` 내에 버튼 전용 변수 및 공통 컴포넌트 레이어 정의
- `DashboardsMain`, `ModalBasic`, `ModalPositionRequestEdit` 등 주요 컴포넌트의 버튼 일괄 리팩토링
- `btn-primary`는 항상 검은색(#000000), `btn-default`는 항상 흰색(#ffffff)으로 고정 (라이트/다크 모드 무관)
- `GENERAL.md` 규칙 업데이트 (커밋 메시지 요청 시점 명시 및 태스크 문서 양식/언어 규정 강화)

## WHY
1. **유지보수 효율성**: 모든 버튼에 `px-4 py-2 bg-black ...` 식으로 유틸리티 클래스를 나열하는 방식은 나중에 디자인을 일괄 변경하기 어렵습니다. `btn-primary` 같은 세맨틱 클래스를 통해 스타일을 중앙 집중화했습니다.
2. **브랜드 아이덴티티**: 기존의 파란색 대신 블랙 & 화이트 테마를 적용하여 더 세련되고 프리미엄한 디자인 룩을 구현하고자 했습니다.
3. **일관된 UX**: 모든 버튼에 동일한 `transition`, `active:scale` 애니메이션, 호버 효과를 적용하여 사용자에게 일관된 인터랙션 피드백을 제공합니다.
4. **프로젝트 관리 표준 준수**: `GENERAL.md`에 명시된 규칙대로 작업 내역을 문서화하고, 특히 "한글 답변" 및 "이전 문서 양식 준수" 규칙을 강화하여 협업의 투명성을 높였습니다.

## HOW & WHY
- **Tailwind v4 대응**: `@apply` 시 동일 레이어 내 클래스 참조 에러(`unknown utility class`)를 방지하기 위해 각 버튼 클래스에 베이스 스타일을 직접 포함(Inline) 배치하여 빌드 안정성을 확보했습니다.
- **Constant Color 적용**: 라이트/다크 모드 전환에 상관없이 특정 버튼의 성격(Primary vs Default)을 유지하기 위해 CSS 변수값을 고정값으로 설정했습니다.
- **Invisible Default Button**: `btn-default`를 흰색 배경에 테두리 없이 설정하여, 라이트 모드에서는 텍스트만 보이다가 호버 시에만 버튼 형태가 드러나는 미니멀한 의도를 반영했습니다.

## RESULT
- 컴포넌트 레벨에서의 스타일 코드량이 대폭 감소하고 가독성이 향상되었습니다.
- 블랙 & 화이트 기반의 차별화된 버튼 UI가 시스템 전반에 적용되었습니다.
- 프로젝트 규칙 문서(`GENERAL.md`)가 보완되어 향후 작업에서도 일관된 언어(한글)와 양식을 유지할 수 있는 근거가 마련되었습니다.
